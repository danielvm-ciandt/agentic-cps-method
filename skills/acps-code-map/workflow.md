# acps-code-map Workflow

**CPS Reference:** CPS Chapter 14 — Architecture and Project Intelligence

## Context

`acps-code-map` builds a structured code map of an existing codebase. It discovers top-level
source directories, spawns one sub-agent (parallel Task tool call) per directory to analyze
that module independently, then merges all findings into `.planning/codebase/` (D-11).
Output is one Markdown file per analyzed module + a top-level `MANIFEST.md` (D-12).

**Parallel analysis (D-11):** One sub-agent per top-level source dir. Each agent independently
analyzes its module: files, exports, dependencies, responsibilities, public interfaces.
**Output (D-12):** `{planningDir}/codebase/{module-slug}.md` per module + `{planningDir}/codebase/MANIFEST.md`

---

<workflow>

<step n="1" goal="Load config and prepare output directory">
  <action>Look for `.acps-config.json` in the project root</action>
  <check if=".acps-config.json exists">
    <action>Read `.acps-config.json` and extract `planningDir` value</action>
  </check>
  <check if=".acps-config.json is missing">
    <action>Set planningDir to `.planning` (silent default)</action>
  </check>
  <action>Create `{planningDir}/codebase/` directory if it does not already exist</action>
</step>

<step n="2" goal="Discover top-level source directories">
  <action>List all top-level entries (files and directories) in the project root.
  Apply the following exclusion list — skip any entry whose name is in:
    `node_modules`, `.git`, `.planning`, `dist`, `build`, `coverage`, `.cache`,
    `.next`, `.nuxt`, `out`, `tmp`, `.turbo`, `.vercel`
  Also skip hidden directories (names starting with `.`) unless they are recognized
  source directories.
  For each remaining entry that is a directory OR a recognized source file directory
  (e.g., `bin/`, `lib/`, `src/`, `packages/`): add to the `modules` list as
  `{ dir_name: "{entry name}", path: "{entry name}" }`.
  </action>
  <check if="modules list is empty">
    <output>No analyzable source directories found. Ensure the project root contains at
    least one source directory (e.g., src/, lib/, bin/) outside the exclusion list.</output>
    <action>HALT</action>
  </check>
</step>

<step n="3" goal="Announce parallel analysis">
  <output>
Analyzing {modules.length} modules in parallel: {module names joined by ", "}.
Spawning one sub-agent per module.
  </output>
</step>

<step n="4" goal="Parallel module analysis — one sub-agent per module (D-11)">
  <action>For EACH module in the modules list, spawn a parallel sub-agent with the
  following analysis instructions. All sub-agents run concurrently; collect all results
  before proceeding to Step 5.

  Sub-agent instructions template (substitute `{module.dir_name}` and `{module.path}`):

  Analyze the `{module.path}` directory. Read every file in the directory (and
  subdirectories) to produce accurate analysis. Return a structured analysis with
  exactly these 7 sections:

- **Module Name:** `{module.dir_name}`
- **Responsibility:** 1-2 sentence description of what this module does and why it exists
- **Files:** Bullet list of all files with a 1-line description each
- **Key Exports / Public Interface:** List of exported functions, classes, CLI commands,
  or API endpoints with brief descriptions (what each one does, not how)
- **Internal Dependencies:** Which other project modules or directories does this module
  import from? List as: `{module name} — reason for dependency`
- **External Dependencies:** Which npm/pip/cargo/go packages does this module use?
  List as: `{package name} — why used`
- **Data Flow:** Brief description of how data enters this module (inputs) and exits it
  (outputs/side-effects). Include file reads/writes if relevant.

</action>
</step>

<step n="5" goal="Write per-module analysis files (D-12)">
  <action>For each collected sub-agent result:
    1. Derive a slug from the module directory name:
       - Replace `/` and `.` with `-`
       - Convert to lowercase
       - Examples: `src/components` → `src-components`, `bin` → `bin`, `lib/utils` → `lib-utils`
    2. Write `{planningDir}/codebase/{slug}.md` with the sub-agent analysis formatted as:

```markdown
## Module: {module.dir_name}

### Responsibility
{responsibility text}

### Files
{files bullet list}

### Key Exports / Public Interface
{exports list}

### Internal Dependencies
{internal deps list, or "None"}

### External Dependencies
{external deps list, or "None"}

### Data Flow
{data flow description}
```

  </action>
</step>

<step n="6" goal="Write MANIFEST.md (D-12)">
  <action>Write `{planningDir}/codebase/MANIFEST.md` with the following structure:

```markdown
# Codebase Manifest

*Generated by acps-code-map on {today's date}*

## Modules

| Module | File | Responsibility | Key Exports |
|--------|------|---------------|-------------|
{one row per module: | {dir_name} | codebase/{slug}.md | {responsibility 1 sentence} | {top 1-3 export names} |}

## Cross-Module Dependencies

{For each module that has internal dependencies, list:}
- **{module name}** imports from: {comma-separated list of modules it depends on}

{If no cross-module dependencies found:}
- No cross-module imports detected.

## Data Flow Overview

{Synthesized high-level description of how data flows between the top modules,
derived from the individual module Data Flow sections. Describe the primary
data path(s) through the codebase: what enters the system, how it moves between
modules, and where it exits or is persisted.}
```

  </action>
</step>

<step n="7" goal="Output completion summary">
  <output>
**acps-code-map complete.**

Modules analyzed: {modules.length}
Output directory: {planningDir}/codebase/

Files written:

{for each module: `{planningDir}/codebase/{slug}.md` ({module.dir_name})}

- {planningDir}/codebase/MANIFEST.md

Review the MANIFEST.md for a full cross-module dependency overview.
  </output>
</step>

</workflow>
