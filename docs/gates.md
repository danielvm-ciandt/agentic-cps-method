# Phase Gates

## Overview

A gate is a checkpoint that must pass before the next phase or iteration
unlocks. Gates enforce the CPS quality discipline: no phase starts until the
previous one is complete and approved. The AI agent runs `/acps-gate` to check
all conditions and either lists blockers or declares the gate open.

## Setup Phase Gate

**Command:** `/acps-gate` (run after setup deliverable is approved)

**Conditions:**

- [ ] `vision.md` written and reviewed
- [ ] `backlog.md` written with all epics and iterations
- [ ] `architecture.md` written
- [ ] `roadmap.md` written with BCP estimates and timeline
- [ ] Setup deliverable generated, reviewed, and sent to client or sponsor
- [ ] Client or sponsor has approved the deliverable

**On gate approval:**

- BCP total from `roadmap.md` is locked as the baseline in `change-log.md`
- Production Flow unlocks: `iteration/1` branch can be created
- Baseline is never modified after this point — only the CR ledger grows

---

## Iteration Gate

**Command:** `/acps-gate` (run at the end of each iteration)

**Conditions:**

- [ ] All epics assigned to this iteration are homologated
- [ ] No open bugs blocking homologation (bugs are registered + fixed + re-homologated)
- [ ] Iteration report generated by `/acps-deliverable`
- [ ] Iteration report reviewed and sent to client
- [ ] Scope ledger is balanced or has an approved overrun (via `/acps-scope-manager`)

**On gate approval:**

- Iteration branch is eligible for merge to main (via `/acps-pr`)
- `/acps-new-iteration` can open the next iteration branch
- Pre-release tag is created by semantic-release on push to main

---

## Value Activation Gate

**Command:** `/acps-gate` (run after value activation is complete)

**Conditions:**

- [ ] Production entry complete — system is accessible to real users
- [ ] Late homologation complete — all epics re-verified with real users
- [ ] All post-release bugs registered, fixed, and re-homologated
- [ ] Value Activation deliverable generated and approved
- [ ] Change management plan in place or explicitly marked N/A

**On gate approval:**

- Project is officially closed
- Final stable release tag is created on main (v1.0.0 or final version)

## What blocks a gate

When `/acps-gate` runs and finds blockers, it outputs:

```text
⛔ Iteration 1 gate — BLOCKED

  [ ] EP-M02 has 3 stories not yet homologated:
      - EP-M02-003: workflow docs — in-progress
      - EP-M02-004: gates doc — backlog
      - EP-M02-005: standards docs — backlog

  [ ] Iteration report not yet generated
      → Run /acps-deliverable

  [ ] 1 open bug: BUG-001 — lint gate failure in EP-M01-001
      → Run /acps-bug-fix

Resolve all blockers and re-run /acps-gate.
```

## Gate approval model

Gates require human approval — the AI cannot approve its own gate. The sequence:

1. AI runs `/acps-gate` → lists all conditions with pass/fail status
2. Human reviews the output and the deliverable
3. Human explicitly approves (types "approve" or equivalent)
4. AI records the approval and unlocks the next phase

The setup gate additionally triggers BCP baseline lock — this is the only
automatic action the AI takes on gate approval.
