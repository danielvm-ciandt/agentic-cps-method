# Setup Phase

## Overview

The Setup Phase establishes the foundation of a new project before any code is
written. It produces four planning artifacts — vision, backlog, architecture, and
roadmap — reviewed and approved by the team before Production Flow begins. The
phase ends with a gate that locks the BCP baseline for scope management.

All artifacts are stored in `.planning/` and are internal — they are not
client-facing deliverables. The client-facing document is generated by
`/acps-deliverable` at gate approval.

## Requirements

- [ ] Product vision drafted and reviewed by stakeholders
- [ ] Full backlog created: all epics named, assigned to iterations
- [ ] Architecture package written: stack decisions, repo structure, ADRs
- [ ] Roadmap written: BCP or FP estimates per epic, iteration timeline
- [ ] Setup deliverable generated and approved by client or sponsor
- [ ] Gate conditions verified before unlocking Production Flow

## Non-goals

- The Setup Phase does not write code
- It does not produce user-facing features
- `.planning/` files are never committed to a client-visible branch

## Acceptance criteria

- [ ] AC 1: `.planning/vision.md` exists with two-repo strategy, 9 CPS practices, and lifecycle diagram
- [ ] AC 2: `.planning/backlog.md` lists all epics with epic IDs, iteration assignments, and priorities
- [ ] AC 3: `.planning/architecture.md` documents tech stack, repo structure, and installer architecture
- [ ] AC 4: `.planning/roadmap.md` includes BCP estimates for all epics and a full timeline
- [ ] AC 5: Gate approved → BCP baseline locked in `change-log.md` → Production Flow unlocks

---

## Commands

```text
/acps-vision           → Ch.4  — guided vision creation → vision.md
/acps-backlog          → Ch.10 — epic and iteration planning → backlog.md
/acps-architecture     → Ch.14 — architecture decisions → architecture.md
/acps-project-roadmap  → Ch.12 — BCP/FP estimates → roadmap.md
/acps-deliverable      → generate setup deliverable for client review
/acps-gate             → verify all gate conditions → lock BCP baseline
```

## Gate checklist

All conditions must be met before the gate passes:

- [ ] `vision.md` written and reviewed
- [ ] `backlog.md` written with all epics and iterations
- [ ] `architecture.md` written
- [ ] `roadmap.md` written with BCP estimates and timeline
- [ ] Client or sponsor has approved the setup deliverable
- [ ] Gate approved → `/acps-gate` locks BCP baseline in `change-log.md`

## Output artifacts

| Artifact | Path | Purpose |
|----------|------|---------|
| Vision | `.planning/vision.md` | Product vision, two-repo strategy, CPS practices |
| Backlog | `.planning/backlog.md` | All epics, stories, iteration assignments |
| Architecture | `.planning/architecture.md` | Technical decisions, repo structure |
| Roadmap | `.planning/roadmap.md` | BCP estimates, iteration timeline |
| Setup deliverable | `deliverables/setup-deliverable.md` | Client-facing summary |
| Change log | `change-log.md` | Baseline BCP locked at gate approval |
