# Production Flow

## Overview

Production Flow is the repeating phase where software is built. It runs as a
series of time-boxed iterations. Each iteration contains one or more epics, each
epic runs through the full GSD loop (branch → discuss → spec → plan → execute →
homologate), and each iteration ends with a gate and a client-facing report.

The inner rhythm mirrors GSD's discipline: branch first, then discuss before
speccing, then spec before planning, then plan before executing. No step is
skipped. The AI agent enforces sequence.

## Requirements

- [ ] `iteration/N` branch created before any epic work begins
- [ ] Each story follows the full GSD loop in order
- [ ] Branch created by `/acps-branch` before any code is written
- [ ] Ambiguity gate passes (score ≤ 0.20) before `/acps-execute` runs
- [ ] All stories homologated before iteration gate is attempted
- [ ] Iteration report generated and sent to client at iteration close

## Non-goals

- Production Flow does not modify scope without a formal change request
- It does not bypass the ambiguity gate — spec quality is non-negotiable
- The iteration branch is not merged to main until the gate passes

## Acceptance criteria

- [ ] AC 1: Every story has a semantic branch created by `/acps-branch` before coding
- [ ] AC 2: Every story passes `/acps-spec` ambiguity gate ≤ 0.20 before `/acps-execute`
- [ ] AC 3: Every story is homologated by `/acps-homologate` before the iteration closes
- [ ] AC 4: Iteration report delivered to client at end of each iteration
- [ ] AC 5: Scope changes go through `/acps-change-request` — no undocumented additions

---

## The iteration loop

```text
/acps-new-iteration   → open iteration/N branch, assign epics

  For each epic:
    /acps-branch        → create feat/ep-XX-N-slug branch
    /acps-discuss       → extract decisions (GSD discuss loop)
    /acps-spec          → acceptance criteria + ambiguity gate ≤ 0.20
    /acps-plan          → tasks + estimates
    /acps-execute       → implementation
    /acps-homologate    → interactive UAT: walk each AC, capture pass/fail

/acps-deliverable     → generate iteration report for client review
/acps-gate            → verify all epics homologated + report sent
/acps-new-iteration   → open next iteration
```

## Scope management within iterations

If new work is discovered during an iteration:

```text
/acps-change-request  → register CR with delta BCP (+ add or − remove)
/acps-scope-manager   → view scope ledger and approve pending CRs
```

Scope rules:

- Adding BCP requires removing equal BCP or getting an approved overrun
- All scope changes are recorded in `change-log.md`
- The BCP baseline set at the setup gate is never modified

## Iteration branch naming

```text
iteration/1        → pre-release: v0.1.0-iter1.N
iteration/2        → pre-release: v0.2.0-iter2.N
```

Feature branches off iteration branches:

```text
feat/ep-m01-1-init-repo-structure
feat/ep-m02-4-phase-docs
fix/12-release-config
```

## Gate checklist (per iteration)

- [ ] All epics in the iteration are homologated
- [ ] Iteration report generated by `/acps-deliverable`
- [ ] Iteration report reviewed and sent to client
- [ ] No open bugs blocking homologation
- [ ] Scope ledger is balanced or has approved overrun
