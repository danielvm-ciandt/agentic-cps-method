Create `skills/acps-bug-fix/workflow.md` with 6 XML steps:

**Step 1: Load config and identify bug** — read `.acps-config.json` for `planningDir`; fall back to `.planning`. Ask: "Which bug are you fixing? Provide the bug ID (e.g., bug-001) or the path to the bug file." Read `{planningDir}/bugs/{bug_id}.md`. If file does not exist: output error and HALT. Extract: bug title, linked epic, linked story, acceptance criteria from the bug file.

**Step 2: Create bug branch** — compose `branch_name = bug/{bug_id}-{slug}` where slug is a kebab-case summary of the bug title (max 5 words, lowercase). Display: "Branch to create: `{branch_name}`". Ask: "Confirm? (Enter to confirm, or type a correction)". If user provides correction: update branch_name. Run `git checkout -b {branch_name}`. If git fails: output error details and HALT.

**Step 3: Implement the fix** — output the bug details as context for implementation:

```
**Bug context for fix — {bug_id}: {bug_title}**

Epic: {epic_id}
Story: {story_id}

Description:
{bug_description}

Steps to Reproduce:
{steps}

Expected: {expected}
Actual: {actual}
```

Ask: "Ready to implement the fix? (yes — I will work through the fix now)" Implement the fix: write code, modify files, run commands as needed to resolve the bug. After fixing, ask: "Fix complete? Describe what was changed."

**Step 4: Verify acceptance criteria** — display the bug's acceptance criteria (from the bug file or the linked story's ## Test section). Ask the user to verify each AC in the running system. Ask: "All ACs verified? (yes / no — describe failures)". If failures reported: note which ACs failed. Ask: "Continue to close bug anyway? (yes / fix more first)". If user needs more time: HALT and tell them to re-run acps-bug-fix when ready.

**Step 5: Close bug file** — update `{planningDir}/bugs/{bug_id}.md`: add `status: closed` and `closed_at: {ISO 8601 timestamp}` to the bug file front-matter or metadata section. Add a `## Resolution` section with a brief description of what was changed (ask user: "One-line summary of the fix?").

**Step 6: Report completion and recommend PR** — output:

```
**acps-bug-fix complete.**

Bug closed: {planningDir}/bugs/{bug_id}.md
Branch: {branch_name}
Status: closed

Commits on this branch:
(run: git log --oneline {base_branch}..{branch_name})

Next step: Run `acps-pr` to create a PR for this bug fix.
```

Create `skills/acps-pr/SKILL.md`:

```markdown
---
name: acps-pr
description: 'Creates a pull request with a conventional commit title. Filters .planning/ commits from the PR body so only code changes appear. Run after implementation is complete on a story or bug branch.'
---

Follow the instructions in ./workflow.md.
```

Create `skills/acps-pr/workflow.md` with 5 XML steps:

**Step 1: Load context and verify git state** — read `.acps-config.json` for `planningDir`; fall back to `.planning`. Run `git branch --show-current` to get current branch name. Run `git status --porcelain` to check for uncommitted changes. If uncommitted changes exist: output "Warning: uncommitted changes detected. Commit or stash before creating a PR." Ask: "Continue anyway? (yes / no)". If no: HALT.

**Step 2: Collect code-only commits (D-18)** — run: `git log --oneline --no-merges origin/main..HEAD -- . ":(exclude).planning/"` to get the list of commits that touch non-planning files. Store as `code_commits`. Also run `git log --oneline --no-merges origin/main..HEAD` for the full commit list (used for counting only). Output a summary showing: "Code commits (will appear in PR): {count}" and list them.

**Step 3: Compose PR title and body** — ask:
- "PR title? Use conventional commit format: `type(scope): description` (e.g., `feat(auth): add token refresh endpoint`)"
- "Additional PR description? (or press Enter to use auto-generated body)"

Build the PR body:

```markdown
## What changed

{code_commits as a bulleted list — each commit message on its own line}

## Story / Bug

{branch name parsed to derive epic-id and story/bug reference}

## Checklist

- [ ] All acceptance criteria verified
- [ ] No .planning/ changes in this PR body (methodology commits excluded)
```

**Step 4: Create the PR** — display the full PR title and body for review. Ask: "Create PR with this content? (yes / edit title / edit body)". If edit: ask for the corrected content. Run `gh pr create --title "{pr_title}" --body "{pr_body}"`. If `gh` CLI is not available: output the PR title and body for the user to paste manually into their git hosting UI.

**Step 5: Report completion** — output:

```
**acps-pr complete.**

PR created: {PR URL or 'see above to create manually'}
Branch: {branch_name}
Title: {pr_title}
Code commits: {count}
Planning commits excluded: {total_commits - code_commits_count}
```
