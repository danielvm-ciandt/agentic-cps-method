  <action>
Write `release.yml` with exactly this content:

```yaml
name: Release

on:
  push:
    branches:
      - main

permissions:
  contents: write
  issues: write
  pull-requests: write

jobs:
  release:
    name: Release
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: actions/setup-node@v4
        with:
          node-version: '24'
          cache: npm
      - run: npm ci
      - run: npx semantic-release
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
```

Critical details:
- `fetch-depth: 0` is MANDATORY — semantic-release needs full git history to compute version bumps and find the previous release tag. Default (fetch-depth: 1) will cause semantic-release to fail or compute wrong versions.
- `permissions` block is required — semantic-release needs `contents: write` to create GitHub Releases and `issues: write` / `pull-requests: write` for linking commits to issues/PRs.
- Both `GITHUB_TOKEN` and `NPM_TOKEN` are in the env block. `NPM_TOKEN` is required even with `npmPublish: false` because `@semantic-release/npm` runs `verifyConditions` regardless of publish setting.
- Trigger is `push` to `main` only — no PR trigger.
- `cache: npm` for consistent dependency installation.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → repo | Workflows run with write permissions; misconfigured permissions could expose secrets or allow unauthorized writes |
| semantic-release → GitHub API | Release job writes tags and releases to the repo using GITHUB_TOKEN |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | .github/workflows/release.yml — permissions | mitigate | Permissions are scoped minimally: contents/issues/pull-requests write only. No packages or admin permissions granted. |
| T-02-02 | Information Disclosure | NPM_TOKEN in release.yml env | mitigate | Use `${{ secrets.NPM_TOKEN }}` — GitHub Actions automatically masks secret values in logs. Never echo or print the token. |
| T-02-03 | Tampering | GitHub Actions supply chain (actions/checkout@v4) | accept | Using @v4 tag matches old repo and architecture doc pattern. SHA pinning deferred per Open Question in RESEARCH.md. |
| T-02-04 | Elevation of Privilege | semantic-release [skip ci] commit | mitigate | `[skip ci]` in release commit message prevents the release workflow from triggering on its own commit, avoiding infinite loops. Already in .releaserc.json message template. |
</threat_model>
