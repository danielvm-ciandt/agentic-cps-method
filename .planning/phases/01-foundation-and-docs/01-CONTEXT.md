<decisions>
## Implementation Decisions

### Package manager
- **D-01:** Use **npm** (not pnpm). No corepack, no pnpm-lock.yaml. CI uses `npm ci`. This diverges from the old repo — do not copy pnpm config files.

### CI/CD structure
- **D-02:** Two separate workflow files: `.github/workflows/ci.yml` (lint on PRs) and `.github/workflows/release.yml` (semantic-release on push to `main`). Do NOT use the old repo's combined single-file approach.
- **D-03:** `ci.yml` triggers on `pull_request` to `main` and `iteration/*` branches. Runs `npm ci`, `npm run lint`, and `node --check bin/install.js`.
- **D-04:** `release.yml` triggers on push to `main`. Runs semantic-release with `GITHUB_TOKEN` and `NPM_TOKEN` secrets.

### Markdownlint tooling
- **D-05:** Use **markdownlint-cli2** (not markdownlint-cli). Lint script: `markdownlint-cli2 "docs/**/*.md" "*.md" "!CHANGELOG.md"`. Can add `.markdownlint.json` config if needed.

### bin/install.js
- **D-06:** Copy stub from old repo verbatim — it exists only to satisfy `node --check` CI gate. Full installer is Phase 4 (EP-M07).

### Methodology docs
- **D-07:** Copy all docs from `/Users/danielvm/Sites/old-agentic-cps-method/docs/` then review and update each file for any content that changed based on the new planning docs (vision.md, architecture.md, backlog.md). Do NOT blindly overwrite without checking for updates.
- **D-08:** Files to copy: `getting-started.md`, `gates.md`, `phases/` (3 files), `practices/` (9 files), `workflow/` (5 files), `standards/` (5 files).

### README
- **D-09:** Write a **fresh README** — do not copy from old repo. Follow the kickass-readme standard: title + one-liner, why it exists, install command (`npx acps@latest`), quick start (5 steps), links to docs. Keep it scannable — not a wall of text.

### .releaserc.json
- **D-10:** Copy from old repo and keep `"npmPublish": false` — GitHub Releases only, no npm registry publish in v1.

### Claude's Discretion
- Exact markdownlint rule configuration (whether to add `.markdownlint.json` or use defaults)
- Node.js version pinning in workflows (>=24 is the constraint, exact version choice is Claude's)
- Branch protection rules documentation in README (recommend but don't configure)

</decisions>
