# acps-change-request Workflow

**CPS Reference:** CPS Chapter 16 — Change Control and Scope Accounting

## Context

`acps-change-request` registers a formal CR entry in `change-log.md`. The AI reads the change description from the user, computes BCP/FP delta from existing story estimates, assigns the next sequential CR ID, and appends the row to the CR table immediately — no confirm step (D-01). If registering this CR causes the running balance to go negative (overrun), a warning banner is shown but the CR is still written (D-04 by extension).

**CR ID format (D-03):** Sequential integers prefixed `CR-` and zero-padded to 3 digits (CR-001, CR-002...). Derived from highest existing row.
**change-log.md format (D-02):** YAML front-matter (baseline_bcp, baseline_fp, approved_at, approved_by) + CR table (CR ID | Date | Description | BCP Delta | FP Delta | Balance)
**Missing file (Claude's Discretion):** If change-log.md does not exist, create it with default YAML front-matter (baseline_bcp: 0, baseline_fp: 0, approved_at: now, approved_by: "default") and an empty CR table before appending.
```

Workflow steps:
- Step 1: Load config — read `.acps-config.json`, extract `planningDir`; default to `.planning` if missing
- Step 2: Read change-log.md — check if `{planningDir}/change-log.md` exists. If missing: create it with YAML front-matter `baseline_bcp: 0`, `baseline_fp: 0`, `approved_at: "{now}"`, `approved_by: "default"` and empty CR table. If exists: read it, extract `baseline_bcp` and `baseline_fp` from YAML front-matter.
- Step 3: Derive next CR ID — scan all existing data rows in the CR table. Find the highest CR number (e.g., if highest is `CR-003`, next is `CR-004`). If no rows exist, next ID is `CR-001`.
- Step 4: Collect CR inputs from user — ask: "Describe the change (1-2 sentences):". Then ask: "BCP delta for this change? (positive = scope addition, negative = removal, 0 = neutral):". Then ask: "FP delta for this change? (positive = new feature points, negative = removal, 0 = neutral):".
- Step 5: Compute running balance — read existing CR table rows, sum all previous BCP Delta values to get previous_bcp_total. Compute: `new_bcp_balance = baseline_bcp + previous_bcp_total + bcp_delta`. Record `cr_date = today's date in YYYY-MM-DD format`.
- Step 6: Append CR row to change-log.md — insert a new row at the end of the CR table:
  `| {cr_id} | {cr_date} | {description} | {bcp_delta with sign, e.g. +5 or -3} | {fp_delta with sign} | {new_bcp_balance} |`
  Write the updated change-log.md.
- Step 7: Check for overrun — if `new_bcp_balance < 0`: display overrun banner: "**SCOPE OVERRUN:** Current BCP balance is {new_bcp_balance}. Baseline: {baseline_bcp}. Gap: {abs(new_bcp_balance)} BCP. Consider removing scope or negotiating baseline adjustment." Do not block — CR is already written.
- Step 8: Output completion message — show CR ID registered, description, BCP delta, FP delta, new balance, path written.

# acps-scope-manager Workflow

**CPS Reference:** CPS Chapter 16 — Change Control and Scope Accounting

## Context

`acps-scope-manager` reads `change-log.md` and renders the full scope ledger: a header block
comparing baseline vs current BCP/FP with balance, followed by the complete CR history table.
On overrun (current > baseline), displays a clear warning banner with the exact gap (D-04).
Information only — does not block work or require action (D-04).

**Ledger display format (D-05):**
- Header block: Baseline BCP | Current BCP | Balance — same row for FP
- CR table: All rows from change-log.md CR table (CR ID, date, description, deltas, running balance)
```

Workflow steps:
- Step 1: Load config — read `.acps-config.json`, extract `planningDir`; default to `.planning` if missing
- Step 2: Check for change-log.md — if `{planningDir}/change-log.md` does not exist: output "No scope baseline found. Run `acps-gate` to lock the BCP baseline and open the change log." then HALT.
- Step 3: Read change-log.md — read YAML front-matter: extract `baseline_bcp`, `baseline_fp`, `approved_at`, `approved_by`. Read CR table: collect all data rows.
- Step 4: Compute scope totals — sum all BCP Delta values from CR rows → `total_bcp_delta`. Sum all FP Delta values → `total_fp_delta`. Compute: `current_bcp = baseline_bcp + total_bcp_delta`. `current_fp = baseline_fp + total_fp_delta`. `bcp_balance = baseline_bcp - current_bcp` (positive = headroom, negative = overrun). `fp_balance = baseline_fp - current_fp`.
- Step 5: Display ledger header block — exact format per D-05:
```
**acps-scope-manager — Scope Ledger**

| Metric | Baseline | Current | Balance |
|--------|----------|---------|---------|
| BCP    | {baseline_bcp} | {current_bcp} | {bcp_balance, show +N green or -N red} |
| FP     | {baseline_fp}  | {current_fp}  | {fp_balance, show +N green or -N red}  |

Approved at: {approved_at} | Approved by: {approved_by}
```
- Step 6: Display CR history table — if no CR rows exist: show "No change requests registered yet. Run `acps-change-request` to register the first CR." If rows exist: display the full CR table with all columns (CR ID | Date | Description | BCP Delta | FP Delta | Balance).
- Step 7: Check for overrun — if `bcp_balance < 0` OR `fp_balance < 0`: display warning banner:
```
**SCOPE OVERRUN DETECTED**
BCP overrun: {abs(bcp_balance)} BCP over baseline (if negative)
FP overrun: {abs(fp_balance)} FP over baseline (if negative)
Consider running `acps-change-request` with a negative delta to remove scope,
or negotiate a baseline adjustment via a new gate approval.
```
If no overrun: display "Scope is within baseline. No action required."
