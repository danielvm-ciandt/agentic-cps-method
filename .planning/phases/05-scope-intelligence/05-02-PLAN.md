```
# acps-document-project Workflow

**CPS Reference:** CPS Chapter 14 — Architecture and Project Intelligence

## Context

`acps-document-project` performs a deep scan of the project codebase and synthesizes 4
documentation files in `{planningDir}/intel/`. On first run it creates all 4 docs. On
re-run it reads `.scan-state.json` to skip completed docs and only regenerate pending ones
(D-07). The state file is deleted when all 4 docs are complete (D-07).

**Output docs (D-06):** project-overview.md, source-tree.md, component-inventory.md, dev-guide.md
**State file (D-08):** `{planningDir}/intel/.scan-state.json` — created on start, deleted on completion
**Output dir:** `{planningDir}/intel/` — created if not present
```

Workflow steps:

- Step 1: Load config — read `.acps-config.json` if present, extract `planningDir`; default `.planning` if missing.

- Step 2: Initialize or resume from state — check if `{planningDir}/intel/.scan-state.json` exists.
  - If exists: read it. Extract `completed` array and `pending` array. Output: "Resuming scan — {completed.length} docs already complete, {pending.length} remaining: {pending.join(', ')}."
  - If missing: create `{planningDir}/intel/` directory if it does not exist. Set `pending = ["project-overview.md", "source-tree.md", "component-inventory.md", "dev-guide.md"]`. Set `completed = []`. Write `.scan-state.json`: `{"started_at": "{now_ISO}", "completed": [], "pending": ["project-overview.md", "source-tree.md", "component-inventory.md", "dev-guide.md"]}`.

- Step 3: Scan project structure — only if `source-tree.md` is in the pending list. Read the directory tree: run `find . -not -path '*/node_modules/*' -not -path '*/.git/*' -not -path '*/dist/*' -not -path '*/.planning/*' -type f | sort` to enumerate all source files. Annotate each directory with a 1-line description based on its contents. Produce an annotated tree structure (directory-level grouping, file listing, purpose annotations).

- Step 4: Scan source code for intelligence — only if any of `project-overview.md`, `component-inventory.md`, or `dev-guide.md` are pending. Read key files in this order:
  - `package.json` or `pyproject.toml` or equivalent manifest — extract: project name, description, main entry, dependencies, scripts
  - `README.md` — extract: purpose statement, install instructions, usage examples
  - Source entry points (e.g., `src/index.*`, `bin/*`, `main.*`) — read to understand architecture
  - Config files (`.env.example`, `tsconfig.json`, `vite.config.*`, etc.) — extract tech stack signals
  - Test files (sample 3-5) — understand test framework and conventions

- Step 5: Write project-overview.md (if in pending) — write `{planningDir}/intel/project-overview.md`:
  Sections: `# Project Overview`, `## What This Project Is` (1-2 paragraph summary), `## Tech Stack` (bulleted list: language, runtime, framework, key dependencies), `## Architecture Summary` (how the system is organized — layers, modules, main data flow). Update `.scan-state.json`: move `project-overview.md` from `pending` to `completed`. Write updated state file.

- Step 6: Write source-tree.md (if in pending) — write `{planningDir}/intel/source-tree.md`:
  Sections: `# Source Tree`, `## Directory Structure` (annotated tree from step 3), `## Key Entry Points` (list of main entry files with 1-line description each). Update `.scan-state.json`.

- Step 7: Write component-inventory.md (if in pending) — write `{planningDir}/intel/component-inventory.md`:
  Sections: `# Component Inventory`, then for each identified major component/module: component name, file path, responsibility (1 sentence), public interface (exported functions/classes/APIs). Organize as a table or heading-per-component. Update `.scan-state.json`.

- Step 8: Write dev-guide.md (if in pending) — write `{planningDir}/intel/dev-guide.md`:
  Sections: `# Developer Guide`, `## How to Install` (extracted from README/package.json), `## How to Run` (extracted from scripts or README), `## How to Test` (test command + framework), `## How to Extend` (patterns for adding new features — derived from code structure). Update `.scan-state.json`.

- Step 9: Finalize — after all 4 docs are written (or confirmed complete from state): delete `{planningDir}/intel/.scan-state.json`. Output completion summary showing the 4 file paths written.

Each step that modifies the state file must update BOTH the `completed` and `pending` arrays atomically and write the JSON file before proceeding to the next document. This ensures crash recovery works correctly.
