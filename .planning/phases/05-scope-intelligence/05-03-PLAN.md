```
# acps-scan Workflow

**CPS Reference:** CPS Chapter 14 — Architecture and Project Intelligence

## Context

`acps-scan` produces a 1-page context summary that orients any AI agent to the project in
under 30 seconds. It reads 6 files + git log directly — no dependency on `acps-document-project`
having run first (D-10).

**5 summary sections (D-09):**
1. What the project is (from package.json + README)
2. Tech stack + key dependencies (from package.json)
3. Current iteration + active epic (from .acps-config.json + backlog.md + git branch)
4. Last 3 git commits (from git log)
5. Open impediments (from notes.md)
```

Workflow steps:

- Step 1: Load config — read `.acps-config.json` if present, extract `planningDir`; default `.planning` if missing.

- Step 2: Read project identity — read `package.json`: extract `name`, `description`, `version`. Read `README.md` (first 50 lines): extract the project's one-line purpose statement. If neither exists, set description to "unknown project".

- Step 3: Read tech stack — read `package.json`: extract `dependencies` and `devDependencies` key counts and top-5 dependency names. Identify: runtime (Node.js/Python/etc from engines or shebang), framework (first framework-like dependency), language (infer from file extensions in root). If no package.json: look for `pyproject.toml`, `Cargo.toml`, `go.mod`, `pom.xml` — extract equivalent data.

- Step 4: Read current iteration and active epic — run `git branch --show-current`. Parse branch name using `iter-{N}/{epic-id}-{slug}` pattern to extract iteration N and epic-id. Read `{planningDir}/backlog.md` (if exists) to get epic title matching epic-id. If no match or no backlog: set iteration to branch name, epic to "unknown".

- Step 5: Read last 3 git commits — run `git log --oneline -3`. Capture the 3 lines (hash + subject). If git is not available or no commits: set to "no commits yet".

- Step 6: Read open impediments — check if `{planningDir}/notes.md` exists. If exists: scan for lines containing `[IMPEDIMENT]`, `[BLOCKER]`, `[OPEN]`, or `- [ ]` (unchecked task items). Collect up to 5 most recent matching lines. If none or file missing: set to "No open impediments found."

- Step 7: Output 1-page summary — exact format:
```
**acps-scan — Project Context Summary**
*Generated: {today's date}*

---

## 1. What This Project Is
{project name} v{version} — {description}

## 2. Tech Stack
- Runtime: {runtime}
- Framework: {framework or "none"}
- Key dependencies: {top 5 dep names}
- Estimated dep count: {total dep count}

## 3. Current Iteration
- Branch: {git branch}
- Iteration: {iter-N or "unknown"}
- Active epic: {epic-id} — {epic title or "unknown"}

## 4. Last 3 Commits
- {hash} {subject}
- {hash} {subject}
- {hash} {subject}

## 5. Open Impediments
{list of open impediment lines, or "None"}

---
*Read via: acps-scan | Source files: package.json, README.md, backlog.md, notes.md, git log*
```

```
# acps-code-map Workflow

**CPS Reference:** CPS Chapter 14 — Architecture and Project Intelligence

## Context

`acps-code-map` builds a structured code map of an existing codebase. It discovers top-level
source directories, spawns one sub-agent (parallel Task tool call) per directory to analyze
that module independently, then merges all findings into `.planning/codebase/` (D-11).
Output is one Markdown file per analyzed module + a top-level `MANIFEST.md` (D-12).

**Parallel analysis (D-11):** One sub-agent per top-level source dir. Each agent independently
analyzes its module: files, exports, dependencies, responsibilities, public interfaces.
**Output (D-12):** `{planningDir}/codebase/{module-slug}.md` per module + `{planningDir}/codebase/MANIFEST.md`
```

Workflow steps:

- Step 1: Load config — read `.acps-config.json` if present, extract `planningDir`; default `.planning` if missing. Create `{planningDir}/codebase/` directory if it does not exist.

- Step 2: Discover top-level source directories — list all top-level directories in the project root, excluding: `node_modules`, `.git`, `.planning`, `dist`, `build`, `coverage`, `.cache`. Also include top-level source files (e.g., `bin/`, `lib/`, `src/`) that are not excluded. Produce a `modules` list: each entry has `{dir_name, path}`.

- Step 3: Announce parallel analysis — output: "Analyzing {modules.length} modules in parallel: {module names joined by ', '}. Spawning one sub-agent per module."

- Step 4: Parallel module analysis (one sub-agent per module, D-11) — for EACH module in the list, spawn a sub-agent with the following instructions:
  "Analyze the `{module.path}` directory. Produce a structured analysis with these sections:
  1. **Module Name:** `{module.dir_name}`
  2. **Responsibility:** 1-2 sentence description of what this module does
  3. **Files:** List all files with 1-line description each
  4. **Key Exports / Public Interface:** List exported functions, classes, or CLI commands with brief descriptions
  5. **Internal Dependencies:** Which other project modules does this import from?
  6. **External Dependencies:** Which npm/pip/cargo packages does this module use?
  7. **Data Flow:** Briefly describe how data enters and exits this module
  Read every file in `{module.path}` to produce accurate analysis."
  Collect all sub-agent results.

- Step 5: Write per-module files (D-12) — for each module result: derive a slug from the directory name (replace `/` and `.` with `-`, lowercase; e.g., `src/components` → `src-components`). Write `{planningDir}/codebase/{slug}.md` with the sub-agent's analysis sections as structured Markdown. Use `## Module: {module.dir_name}` as the H2 heading, then the 7 analysis sections.

- Step 6: Write MANIFEST.md (D-12) — write `{planningDir}/codebase/MANIFEST.md` with:
  - `# Codebase Manifest` header
  - `## Modules` section: table with columns `Module | File | Responsibility | Key Exports`; one row per analyzed module
  - `## Cross-Module Dependencies` section: a dependency matrix or bulleted list showing which modules import from which (derived from the "Internal Dependencies" field of each module analysis)
  - `## Data Flow Overview` section: high-level description of how data flows between the top modules, synthesized from individual module data flows

- Step 7: Output completion summary — list all files written in `{planningDir}/codebase/`, total module count, and path to MANIFEST.md.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| filesystem → acps-scan | Reads package.json, README, backlog.md, notes.md — all local project files |
| filesystem → acps-code-map sub-agents | Sub-agents read arbitrary source files; no user-controlled exec paths |
| acps-code-map → .planning/codebase/ | Writes synthesized markdown; no execution of analyzed content |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-03-01 | Information Disclosure | acps-scan output (git log, tech stack) | accept | Output is local context summary; not transmitted. Low risk, high utility for AI agents. |
| T-05-03-02 | Information Disclosure | acps-code-map codebase/ output | accept | Contains codebase analysis stored locally in .planning/codebase/ under version control. No remote surface. |
| T-05-03-03 | Denial of Service | acps-code-map with very large module trees | accept | Sub-agent spawning is bounded by top-level directory count (typically <20). AI context window is natural limit. |
| T-05-03-04 | Tampering | notes.md content injected into acps-scan output | accept | notes.md is a local project file read and displayed verbatim; trusted source. No shell execution. |
| T-05-03-05 | Spoofing | git log output spoofed by malicious commit subjects | accept | Local git repository; attacker would need repository write access. Out of scope for v1. |
</threat_model>
